### Общая структура: world_v1.urdf

```xml
<?xml version="1.0"?>
<sdf version="1.10">
    <world name="car_world">
```

*   **`<sdf version="1.10">`** — корневой элемент. 
*   **`<world name="car_world">`** — объявляет новый мир с логическим именем `car_world`. Всё, что находится внутри этого тега, — это население мира: физические законы, освещение, объекты и, конечно, наш будущий робот.

### 2. Плагины симуляции — «Операционная система» мира

Первыми внутри `<world>` идут плагины. В Gazebo Sim логика симуляции строится из систем-плагинов, необходимо явно указать, какие из них нужны.

```xml
<plugin filename="gz-sim-sensors-system" name="gz::sim::systems::Sensors">
    <render_engine>ogre2</render_engine>
</plugin>
```
*   **`Sensors`** — ключевой плагин, отвечающий за рендеринг всех сенсоров: камер, лидаров, глубин и т.д.
*   `render_engine` — указано `ogre2`, высокопроизводительный движок визуализации, который будет использоваться для расчёта изображений с камеры и для **GPU-лидара**.

```xml
<physics name="1ms" type="ignored">
    <max_step_size>0.001</max_step_size>
    <real_time_factor>1.0</real_time_factor>
</physics>
```
*   **`Physics`** — это не плагин в привычном смысле, а настройка физического движка.
*   `max_step_size="0.001"` — максимальный шаг симуляции (1 мс). Это очень маленький шаг, обеспечивающий высокую точность физических расчётов и стабильность, особенно важную для колёсных роботов.
*   `real_time_factor="1.0"` — симуляция будет стараться идти в реальном времени (1 секунда в симуляции = 1 секунда на стене). Значение игнорируется (`type="ignored"`), если движок не может удержать темп, но это наша цель.

```xml
<plugin filename="gz-sim-physics-system" name="...Physics"/>
<plugin filename="gz-sim-user-commands-system" name="...UserCommands"/>
<plugin filename="gz-sim-scene-broadcaster-system" name="...SceneBroadcaster"/>
```
*   **`Physics`-плагин** (отдельно от настроек выше) — непосредственно запускает физический движок.
*   **`UserCommands`** — позволяет нам управлять симуляцией извне (ставить на паузу, перемещать объекты мышкой в GUI).
*   **`SceneBroadcaster`** — передаёт информацию о сцене другим инструментам, например, для визуализации в ROS 2 через `ros_gz_bridge`.

### 3. Освещение: Имитация солнца

```xml
<light type="directional" name="sun">
    <cast_shadows>true</cast_shadows>
    <pose>0 0 10 0 0 0</pose>
    <diffuse>0.5 0.5 0.5 1</diffuse>
    <specular>0.2 0.2 0.2 1</specular>
    <attenuation>...</attenuation>
    <direction>-0.5 0.1 -0.9</direction>
</light>
```

*   **Тип `directional`** — источник света, лучи которого параллельны, как от очень далёкого источника. Идеально для имитации солнца.
*   **`pose`** — позиция источника в пространстве. Высоко над центром мира (10 м по Z).
*   **`direction`** — вектор направления лучей (`-0.5, 0.1, -0.9`). Свет падает немного слева и спереди, создавая естественное освещение.
*   **`cast_shadows=true`** — объекты будут отбрасывать тени. Это критически важно для реалистичности работы камеры и лидара (луч лидара должен останавливаться об объект, а не проходить сквозь тень).
*   **`diffuse` и `specular`** — компоненты освещения (рассеянный и зеркальный). Здесь они приглушённые, создавая мягкую картинку без пересветов.
*   **`attenuation`** — затухание света с расстоянием. Благодаря ему объекты на разном удалении освещены равномерно, без излишнего падения яркости.

### 4. Грунт: Бесконечная плоскость

```xml
<model name="ground_plane">
    <static>true</static>
    <link name="link">
        <collision name="collision">
            <geometry>
                <plane><normal>0 0 1</normal></plane>
            </geometry>
        </collision>
        <visual name="visual">
            <geometry>
                <plane><normal>0 0 1</normal><size>100 100</size></plane>
            </geometry>
            <material>
                <ambient>0.8 0.8 0.8 1</ambient>
                <diffuse>0.8 0.8 0.8 1</diffuse>
                <specular>0.8 0.8 0.8 1</specular>
            </material>
        </visual>
    </link>
</model>
```

*   **Модель `ground_plane`** — это пол. Атрибут **`static="true"`** означает, что объект намертво вморожен в пространство и не будет сдвигаться под весом робота. Физический движок не будет рассчитывать для него динамику, что экономит ресурсы.
*   **Геометрия `plane`** с нормалью `0 0 1` (ось Z) — это классический способ задать бесконечный пол на высоте Z=0.
*   **Коллизия** — простая плоскость (без размеров, она бесконечна для физики). Важно, чтобы робот не провалился сквозь неё.
*   **Визуал** — плоскость размером 100×100 м. Это «текстурированная» область, которую мы видим в симуляторе.
*   **Материал** — нейтральный светло-серый (`0.8 0.8 0.8`) с одинаковыми ambient, diffuse, specular компонентами. Это даёт матовое покрытие, на котором хорошо видны тени и другие объекты.

###  Итог



1.  **Физика настроена на точность** (шаг 0.001 с) — колёсный робот будет катиться плавно, без артефактов.
2.  **GPU-лидар и камера с тенями** — плагин рендеринга `ogre2` и солнце, отбрасывающее тени, гарантируют, что лидар получит реалистичные облака точек, а камера — детализированную картинку с правильным освещением.
3.  **Разноцветные кубы** — это эталонные мишени. Выровненные в линию, они образуют «лесенку» на расстоянии от 2 до 8 метров.
    *   **Для лидара** это семь чётко различимых препятствий. Анализируя отражённую интенсивность, можно определять, какой куб какого цвета, или просто видеть их геометрию.
    *   **Для камеры** это идеальный тест цветопередачи. Разработчик может запустить робота и проверить, корректно ли работает алгоритм распознавания цвета, видит ли он последовательность «радуги».
4.  **Статичный пол** и отсутствие лишних объектов — мир максимально лёгкий, быстро загружается и не нагружает симуляцию, позволяя сосредоточиться именно на алгоритмах управления и восприятия робота.
